# Seldon Paper Skill

Research paper writing with full result provenance, terminology control, and
graph-backed traceability. Every number in the paper traces to a script, data
file, and requirement. Every term has a controlled definition. Every claim maps
to registered evidence.

## Project Setup

The paper project must have Seldon initialized. Check with:
```bash
seldon status
```

If not initialized: `seldon init --domain research`

### Required Files

These live in the project's `paper/` directory:

| File | Purpose | How Created |
|------|---------|-------------|
| `paper/glossary.md` | Controlled vocabulary + banned synonyms | Manual — one entry per term |
| `paper/check_glossary.py` | Term validator + keyword index generator | Template (will be `seldon paper init`) |
| `paper/keyword_index.md` | Auto-generated concordance | Generated by `check_glossary.py` |
| `paper/evidence_map.md` | Results → claims → sections mapping | Manual — updated when results/claims change |
| `paper/conventions.md` | Style rules, formatting rules | Manual — project-specific |
| `paper/sections/NN_name.md` | Section files (numbered, sorted) | Manual — the actual writing |

### Conventions (Read Before Any Edit)

- No em dashes. Zero tolerance.
- No inline bold in prose.
- Max 35 words per sentence.
- Min 2 sentences per paragraph.
- "We" throughout, active voice preferred.
- No "novel", "robust", "leverage", "utilize".
- No throat-clearing ("It is worth noting...").
- All research numbers use `{{result:NAME:value}}` syntax — NEVER literal numbers for measured/computed values.
- Experimental parameters (population sizes, seed values, terminal counts) are acceptable as literals.
- Read the project's `paper/conventions.md` for the full list.

## Core Workflows

### The Edit Cycle (MANDATORY after every change to section files)

```bash
python paper/check_glossary.py       # validate terms, regenerate keyword index
seldon paper sync                    # reconcile graph state with files on disk
seldon paper build --no-render       # verify all references resolve
```

All three must pass clean. If the glossary checker finds violations, fix them
before proceeding. If sync finds stale sections, that's expected after edits.
If build fails on unresolved references, a result is missing or misnamed.

**When the user says:** "check it", "run the checks", "verify", "sync",
"does it build" → run this cycle.

### Register a Result

```bash
seldon result register \
  --name descriptive_snake_case_name \
  --value 42.7 \
  --units "seconds" \
  --description "Mean runtime across 5 seeds" \
  --script path/to/generating_script.py \
  --data path/to/input_data.json
```

Then reference it in section prose as `{{result:descriptive_snake_case_name:value}}`.

**Naming convention:** lowercase, underscores, descriptive. Examples:
`logprec_minimal_5_5`, `wrong_limit_ti_15_93`, `scaling_grid_t4_p1000`.

**When the user says:** "register this number", "that should be a result",
"is X registered", "how many results do we have" → use these commands.

### Look Up Results

```bash
seldon result list                    # all registered results
seldon result show <name>             # details + provenance for one result
seldon result trace <name>            # full provenance chain
```

### Verify a Result

```bash
seldon result verify <name>           # re-run generating script, compare
```

If the output matches: state → verified. If mismatch: flags as stale,
propagates staleness to all downstream citations.

### Audit a Section

There is no `seldon paper audit` command yet. Do this manually:

1. Read the section file
2. List all `{{result:...}}` references — verify each resolves via `seldon result list`
3. Find all literal numbers — classify each as:
   - **Acceptable literal** (experimental parameter, structural count, math constant)
   - **Should be registered** (measured value, computed value, derived bound)
4. Cross-reference against `paper/evidence_map.md` — are all claims backed by evidence?
5. Check for terminology violations: `python paper/check_glossary.py`
6. Write findings to `paper/audit_NN_sectionname_YYYY-MM-DD.md`

**When the user says:** "audit section 5", "are there unregistered numbers",
"check the evidence map", "what's missing" → do this workflow.

### Paper Sync (Bidirectional)

```bash
seldon paper sync                     # full sync: hashes + edges
seldon paper sync --dry-run           # show what would change, don't change it
seldon paper sync --auto-stale        # auto-transition modified sections to stale
```

What sync does:
- Computes SHA-256 content hashes on section files
- Compares to stored hashes on PaperSection artifacts in graph
- Reconciles `cites` edges based on current `{{result:NAME:value}}` references in files
- Removes edges for references that were deleted from sections
- Adds edges for references that were added to sections
- Optionally transitions modified sections to `stale` state

### Paper Build

```bash
seldon paper build                    # full build with Quarto render
seldon paper build --no-render        # structural check only (faster)
```

Build resolves all `{{result:NAME:value}}` references to actual values,
runs QC tiers (Tier 1: structural, Tier 2: prose rules, Tier 3: style),
and optionally renders via Quarto.

**Tier 1 violations are blockers** — unresolved references, broken links.
Tier 2 and Tier 3 are warnings.

### Register Section Files

```bash
seldon paper register --all           # register all sections in paper/sections/
seldon paper register <filepath>      # register a single section
```

Creates PaperSection artifacts with initial content hashes.

## Terminology Management

### Glossary Format

Each term in `paper/glossary.md` follows this pattern:

```markdown
**Term Name**
: Definition of the term in this paper's context.
: Usage notes if needed.
: Do not write: "banned synonym 1", "banned synonym 2".
```

The "Do not write:" line is machine-parsed by `check_glossary.py`.
Quoted phrases on that line are flagged as violations if found in any section.

### Adding a New Term

1. Add the entry to `paper/glossary.md` with definition and any banned synonyms
2. Run `python paper/check_glossary.py` to verify no existing violations
3. If violations found in existing sections, fix them

### Renaming a Term

1. Update the glossary entry with the new name
2. Add the old name to the "Do not write:" line
3. Run `python paper/check_glossary.py` — it will flag all old-name occurrences
4. Fix all flagged occurrences in sections
5. Run the full edit cycle (check → sync → build)

### EXEMPT Lines

Some terms have context-dependent usage. A section line can be marked exempt:

```markdown
<!-- EXEMPT: diversity --> We use diversity injection to maintain population variety.
```

The checker skips that line for the specified banned term.

## Evidence Map

`paper/evidence_map.md` maps the argument structure:

- **Results Used in Paper** — table of all registered results, which sections use them, what claim each supports
- **Key Figures/Tables** — what generates each, from what data
- **Claims → Evidence** — for each claim the paper makes, what result or figure backs it

**This is currently a manually maintained document.** Update it when:
- A new result is registered
- A result is assigned to a different section
- A claim is added, removed, or moved between sections
- A figure or table is created or regenerated

**When the user says:** "update the evidence map", "what claims does section 5 make",
"is this claim backed by evidence", "show me the provenance" → consult this file
and the graph.

## Writing Order

Write backwards from the argument:

1. **Conclusion** — what do we claim?
2. **Results** — what evidence supports the claims?
3. **Experimental Design** — how did we produce the evidence?
4. **Methods** — what machinery did we use?
5. **Background** — what context does the reader need?
6. **Introduction** — what question are we answering?
7. **Abstract** — compress everything (revise last)

Each layer constrains the next. The conclusion constrains which results matter.

## CC Task Patterns

### Writing a CC Task for Section Editing

CC tasks that modify section files MUST include:
1. Instruction to read `paper/glossary.md` and `paper/conventions.md` first
2. The edit instructions
3. Mandatory post-edit cycle: `check_glossary → sync → build`
4. Clear separation of mechanical fixes (CC can do) vs judgment calls (flag for human)

### Writing a CC Task for Result Registration

CC tasks that register results MUST include:
1. The exact values, units, and descriptions
2. Provenance: which script generated it, from what data
3. Instruction to update `paper/evidence_map.md`
4. Post-registration: `sync → build` to verify references resolve

## What Seldon Paper Does NOT Have Yet

These are identified gaps from dogfooding. They may become Seldon commands:

| Gap | Current Workaround |
|-----|-------------------|
| `seldon paper audit` — automated literal detection + registry cross-ref | Manual audit workflow (see above) |

| Evidence map as graph query (`seldon paper coverage`) | Manually maintained markdown |
| `seldon paper glossary check` — glossary checker as Seldon command | Standalone `check_glossary.py` |
| Literal number detection in sync | Manual grep |
